# 07. Offline Sync

## Цель

Приложение должно работать как PWA с полноценным офлайн-режимом и последующей синхронизацией.

Офлайн-данные считаются неподтверждёнными, пока сервер их не проверит.

## Главный риск

Если 3 АЗС работают офлайн, возможен конфликт.

Пример:

1. Машина заправилась на АЗС №1.
2. Интернет пропал.
3. На АЗС №2 эту же машину тоже пропустили офлайн.
4. После восстановления интернета сервер видит повторную заправку.

Поэтому система должна показывать конфликт.

## Локальное хранилище

Используем:

- IndexedDB;
- Dexie.

Локальные таблицы:

```text
local_profiles
local_stations
local_daily_limits
local_reservations
local_queue_entries
local_fueling_records
local_refusal_records
local_manual_overrides
sync_outbox
sync_conflicts
```

## Sync outbox

Каждая офлайн-операция записывается в `sync_outbox`.

Поля:

```text
id
client_mutation_id
type
payload
status
created_at
synced_at
error
retry_count
```

## Типы операций

```text
CREATE_RESERVATION
UPDATE_RESERVATION_STATUS
CREATE_QUEUE_ENTRY
UPDATE_QUEUE_STATUS
CREATE_FUELING_RECORD
CREATE_REFUSAL_RECORD
CREATE_MANUAL_OVERRIDE
CREATE_DAILY_LIMIT
UPDATE_DAILY_LIMIT
CREATE_RESERVATION_CALL_LOG
```

## Статусы синхронизации

```text
PENDING
SYNCING
SYNCED
FAILED
CONFLICT
```

## client_mutation_id

Каждая операция должна иметь `client_mutation_id`.

Это нужно, чтобы:

- не создать дубль при повторной отправке;
- безопасно повторять запрос;
- понимать, какая локальная операция связана с серверной записью.

## Что можно делать офлайн

Можно:

- проверить автомобиль по локальной базе;
- создать предварительную запись локально;
- добавить в локальную очередь;
- зафиксировать отказ;
- зафиксировать факт заправки как неподтверждённый;
- посмотреть ранее загруженные данные.

Интерфейс обязан показать:

```text
Режим офлайн. Данные будут проверены после синхронизации.
```

## Что нельзя считать окончательным офлайн

Нельзя считать окончательно подтверждённым:

- разрешение на заправку;
- создание записи;
- фиксацию факта заправки;
- остаток лимита;
- отсутствие повторной заправки по всем 3 АЗС.

## Синхронизация

Когда интернет вернулся:

1. Проверить Supabase session.
2. Загрузить профиль пользователя.
3. Взять операции из `sync_outbox`.
4. Отправлять по порядку создания.
5. Каждую операцию отправлять через `sync_offline_mutation`.
6. Сервер проверяет бизнес-правила.
7. Успешную операцию пометить `SYNCED`.
8. Конфликтную операцию пометить `CONFLICT`.
9. Обновить локальную базу с сервера.

## Конфликты

Конфликт возникает, если:

- автомобиль уже заправлен;
- нет активной записи;
- лимит закончился;
- запись отменена;
- запись изменилась;
- пользователь потерял права;
- машина заблокирована;
- ручное разрешение уже использовано;
- АЗС не совпадает.
- успешный звонок был сохранён офлайн, но запись больше не входит в актуальный список обзвона.

## Очередь и топливо

Локальная очередь хранит серверные подсказки:

```text
fuel_preference_mode
is_callable_now
call_unavailable_reason
matched_fuel_type
```

Эти поля нужны для UX в offline-режиме, но не являются окончательным разрешением. При синхронизации сервер заново вычисляет совместимость топлива, дневной лимит и возможность успешного звонка.

## Кто решает конфликты

Конфликты решает:

- mayor;
- station_manager.

Кассир и помощник мэра не решают конфликт самостоятельно.

## UI для синхронизации

Показывать:

- онлайн/офлайн статус;
- количество ожидающих операций;
- количество ошибок;
- количество конфликтов;
- кнопку повторной синхронизации;
- список конфликтных операций.
