# AGENTS.md

Файл правил для ИИ-ассистентов, Cursor, Codex и других агентов разработки.

## Контекст

Проект — мобильное PWA-приложение для контроля очереди и отпуска топлива на 3 АЗС.

Сценарии:

- предварительная запись автомобиля на дату;
- проверка автомобиля по госномеру;
- допуск или запрет к заправке;
- фиксация факта заправки;
- контроль лимитов;
- офлайн-работа с синхронизацией;
- отчёты для администрации.

## Стек

Использовать:

- Vite, React, TypeScript;
- React Router;
- Tailwind CSS;
- shadcn/ui;
- React Hook Form;
- Zod;
- Supabase Auth;
- Supabase PostgreSQL;
- Supabase RPC;
- TanStack Query;
- Zustand;
- Dexie;
- IndexedDB;
- vite-plugin-pwa.

Не заменять стек без отдельного решения.

## Архитектура

Проект строится по FSD.

Слои:

```text
app
pages
widgets
features
entities
shared
```

Разрешённое направление зависимостей:

```text
app -> pages -> widgets -> features -> entities -> shared
```

Запрещено:

- писать бизнес-логику в `pages`;
- обращаться к Supabase напрямую из UI-компонентов;
- смешивать entities и features;
- импортировать внутренности чужих slices вместо public API через `index.ts`;
- создавать общие компоненты внутри конкретной страницы;
- обходить Zod-валидацию в формах;
- создавать критичные записи напрямую через `insert`, если для этого есть RPC.

## Бизнес-правила

Перед изменением логики проверки, записи, заправки, лимитов или синхронизации обязательно смотреть:

- `docs/04-business-rules.md`
- `docs/07-offline-sync.md`
- `docs/11-supabase-rls-rpc.md`

Главное правило:

> Один обычный автомобиль может быть заправлен только один раз в календарный день по всем 3 АЗС.

## Supabase

Использовать Supabase Auth для входа.

Бизнес-роли хранить в `profiles`.

Критичные операции выполнять через RPC:

- `check_vehicle_access`
- `create_reservation`
- `create_fueling_record`
- `create_daily_limit`
- `sync_offline_mutation`
- `create_manual_override`

UI-компоненты, страницы и widgets не должны импортировать Supabase client напрямую. Доступ к данным должен идти через feature/model/api или RPC wrappers в `shared/api/rpc`.

## Public API

Каждый slice в `pages`, `widgets`, `features` и `entities` экспортирует наружу только через `index.ts`.

Разрешено импортировать внутренние файлы slice только из этого же slice. Из других slices использовать public API:

```ts
import { CheckVehicleForm } from '@/features/check-vehicle'
```

Не использовать:

```ts
import { schema } from '@/features/check-vehicle/model/schema'
```

## Формы

Все формы должны использовать:

- React Hook Form;
- Zod;
- `@hookform/resolvers/zod`;
- shadcn/ui Form.

## Переиспользование компонентов

Перед созданием или изменением UI-компонента обязательно сначала провести инвентаризацию существующего UI:

- проверить `src/shared/ui`;
- проверить `src/widgets/*/ui`, если нужен крупный блок интерфейса;
- проверить `src/features/*/ui`, если нужен компонент для действия пользователя;
- проверить `src/entities/*/ui`, если нужен компонент бизнес-сущности;
- проверить публичные экспорты через `index.ts`;
- проверить shadcn/ui-компоненты, уже подключённые в проекте.

Если похожий компонент уже есть, сначала адаптировать его через props, variants, composition или небольшое расширение API. Не копировать компонент в новый slice и не создавать визуально похожий дубль.

Новый компонент допустим только если:

- существующие компоненты не покрывают сценарий даже после небольшого расширения;
- расширение существующего компонента ухудшит его назначение или сломает текущие места использования;
- компонент размещён на самом низком подходящем FSD-слое, чтобы его можно было переиспользовать.

При изменении переиспользуемого компонента сохранять обратную совместимость public API или явно обновлять все места использования.

## Офлайн

Офлайн-операции не считаются подтверждёнными до успешной синхронизации.

Любая офлайн-операция должна иметь:

- `client_mutation_id`;
- `sync_status`;
- запись в `sync_outbox`.

При конфликте операция получает статус `CONFLICT`.

Если новое действие может работать без интернета, перед реализацией определить:

- что сохраняется локально;
- какой `client_mutation_id` используется;
- какой `sync_status` получает запись;
- какая операция попадает в `sync_outbox`;
- что происходит при `CONFLICT`.

## Зависимости

Не добавлять новые production dependencies без явной причины. Сначала проверить текущий стек, `shared/ui`, `shared/lib`, shadcn/ui, TanStack Query, Zustand, Dexie, date-fns и стандартные Web API.

Новая зависимость допустима только если она закрывает реальную сложность, не дублирует текущий стек и её необходимость можно объяснить в итоговом ответе или PR.

## Проверка изменений

После изменения кода запускать минимально релевантные проверки:

- `npm run typecheck`;
- `npm run lint`;
- `npm run check:architecture`;
- `npm run test`, если менялись бизнес-логика, формы, helpers, schemas или offline sync;
- `npm run build` перед финальной проверкой крупных изменений.

## Перед созданием нового кода

1. Определи слой FSD.
2. Проверь, есть ли похожая entity/feature/widget.
3. Если задача затрагивает UI, проверь существующие компоненты и варианты переиспользования.
4. Проверь public API и импортные границы.
5. Проверь бизнес-правила.
6. Проверь роли и права.
7. Проверь офлайн-поведение.
8. Добавь типы и Zod-схемы.
9. Не нарушай импортные границы.

## Если есть сомнение

- бизнес-сущность — `entities`;
- действие пользователя — `features`;
- крупный блок UI — `widgets`;
- страница маршрута — `pages`;
- базовый UI, утилита, config — `shared`;
- провайдеры и роутер — `app`.
